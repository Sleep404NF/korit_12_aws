# AWS 시작하기

---

## 1. 클라우드 컴퓨팅(Cloud Computing)

데스크탑 컴퓨터를 사용하려면 본체 전원이 필요하듯이, 웹 서비스를 운영하고 있다면 서비스를 실행하고 있는 컴퓨터의 본체를 관리해야 한다. 본체가 꺼지게 된다면 사용자가 웹 사이트에 접속하지 못하게 된다.

이 문제를 해결하기 위한 개념이 클라우드 컴퓨팅이다.  
**서버, 스토리지, 데이터베이스 등 컴퓨터 자원을 필요한 만큼만 원격으로 빌려서 사용할 수 있도록 하는 기술**을 의미한다.

> 대표적인 클라우드 서비스: AWS, MS Azure, GCP  
> 수업은 가장 사용률이 높은 AWS 기준으로 진행한다.

---

## 2. AWS란?

처음에는 웹 사이트 호스팅 정도로만 쓰였지만, 오늘날에는 **DB, Storage, AI 등** 다양한 서비스를 제공한다.  
기업은 AWS에서 컴퓨터 자원을 빌려서 사용하고, 사용량에 따라 비용을 지불하는 형태로 운영비용을 절감할 수 있다.

### AWS를 따로 학습하는 이유

1. **시장 점유율** — 2024년 기준 국내 기업의 약 60%가 AWS 사용 (과학기술정보통신부 조사)
2. **다양한 서비스와 통합 옵션** — 컴퓨팅, 스토리지, DB, AI, 머신러닝 등 쉽게 연동 가능
3. **글로벌 인프라** — 전세계적으로 사용 가능

---

## 3. AWS 리전(Region)

`AWS가 전 세계에서 데이터센터를 클러스터링하는 물리적 위치`  
쉽게 말하면 **AWS가 컴퓨터들을 실제로 설치해둔 위치**

### 특징
- 전 세계적으로 다양한 리전 보유, 사용자가 직접 선택 가능
- 리전마다 고유 코드 배정 (예: `ap-northeast-2` = 서울)
- **주의: 기본값이 버지니아 북부(`us-east-1`)로 설정되어 있으므로 반드시 리전을 변경해야 한다**

### 리전 선택 기준

서비스를 주로 사용하는 사용자들과 가까운 위치로 선택하는 것이 베스트.

> 한국 사용자를 대상으로 하는 서비스인데 백엔드가 버지니아 북부, DB가 오사카에 있다면  
> 부산 → 버지니아 북부 → 오사카 → 버지니아 북부 → 부산 순으로 신호가 오가게 된다.  
> 물리적 거리가 멀수록 응답 속도(핑)가 느려진다.

### 수업에서 다루는 AWS 서비스

| 서비스 | 역할 |
|--------|------|
| EC2 | 가상 서버 (백엔드 배포) |
| Route53 | DNS 관리 (유료로 인해 생략 가능) |
| ELB (Elastic Load Balancer) | 트래픽 분산 |
| RDS | 관리형 DB 서버 |
| S3 | 파일/정적 파일 저장소 (프론트엔드 배포) |
| CloudFront | CDN (S3 앞단에서 빠른 배포) |

## 4. 필수 개념

### 배포(Deployment)

**구현한 서비스를 서버나 클라우드 환경에 업로드**하는 것.  
배포 전에는 `localhost`에서만 접근 가능하지만, 배포 후에는 인터넷을 통해 누구나 접근할 수 있다.

EC2에 배포하면 **퍼블릭 IP 주소**를 획득하게 되고, 사용자는 이 IP로 접속할 수 있다.

### IP 주소

인터넷에서 **특정 컴퓨터를 가리키는 주소**.  
모든 컴퓨터에 IP 주소가 부여되어 있다.

```
13.125.0.1  ← 숫자와 점으로 이루어진 형식 (각 자리: 0~255)
```

### 포트(Port)

**한 컴퓨터 내에서 실행되는 특정 프로그램의 주소**

```
13.125.0.1:8080  → 해당 컴퓨터에서 8080 포트로 실행 중인 프로그램 (보통 Spring Boot)
13.125.0.1:5173  → 해당 컴퓨터에서 5173 포트로 실행 중인 프로그램 (Vite/React)
```

> `localhost`는 자기 컴퓨터 IP 주소를 매번 외우기 힘드니까  
> 환경변수로 `127.0.0.1`에 할당해둔 별칭이다.

### Well-Known Port (잘 알려진 포트)

포트는 0~65535까지 있으며, 0~1023번은 특정 용도로 권장되어 있다.

| 포트 | 프로토콜 | 설명 |
|------|---------|------|
| `22` | SSH | EC2 원격 접속 시 사용. Secure Shell의 축약어 |
| `80` | HTTP | 포트 번호 미입력 시 기본 통신 포트 |
| `443` | HTTPS | 보안 HTTP 통신 포트 |

> 권장 사항이지 필수는 아니다. 의도적으로 다른 포트를 사용해서 잠재적 공격을 줄이는 경우도 있다.

## 5. EC2

### 정의
원격으로 접속해 사용할 수 있는 **가상 컴퓨터를 빌려주는 서비스**.

### EC2 인스턴스 구성 요소

| 항목 | 설명 |
|------|------|
| OS 이미지 | EC2에 탑재된 운영체제. Windows, Mac, Ubuntu 등 선택 가능. 가장 가볍게 쓰려면 Ubuntu |
| 인스턴스 유형 | 컴퓨터 사양 (CPU, 메모리 등) |
| 스토리지 | 파일 저장 용량. Spring Boot 프로젝트를 돌릴 용량 필요 |

### EC2 사용 이유
1. **관리 부담 절감** — 정전 등 예상치 못한 상황에서도 AWS가 관리
2. **쉬운 보안 설정** — AWS가 템플릿 형태로 보안 설정을 제공

---

## 6. EC2 세팅 순서

1. 이름 및 태그 설정 후 애플리케이션 OS 이미지 선택 (Ubuntu 권장)
2. 인스턴스 유형 설정
3. 키 페어 생성 및 저장 (`.pem` 파일 — 분실 시 재발급 필요)
4. 설정한 키 페어 선택 후 인스턴스 시작

## 7. 보안 그룹 (Security Group)

EC2 인스턴스를 집이라고 가정하면, 보안 그룹은 **집 바깥의 울타리 및 대문**에 해당한다.  
인스턴스 내부로 들어오기 전에 해당 요청이 허용된 요청인지 검사한다.

### Inbound / Outbound

| 구분 | 방향 |
|------|------|
| Inbound | 외부 → EC2 인스턴스로 들어오는 트래픽 |
| Outbound | EC2 인스턴스 → 외부로 나가는 트래픽 |

### 보안 그룹 설정 (이번 수업 기준)

| 포트 | 용도 |
|------|------|
| `22` | EC2 원격 접속 (SSH) |
| `80` | HTTP 요청 (백엔드 서버) |

> Spring Boot 기본 포트는 8080이지만, `application.yml`에서 80으로 변경하여 배포한다.  
> 80 포트는 HTTP 기본 포트이므로 URL에 포트 번호를 따로 입력하지 않아도 된다.

## 8. EC2에 백엔드 서버 올리기

### 전체 흐름

```
EC2 인스턴스 생성
  → 보안 그룹 설정 (22, 80 포트 오픈)
    → 터미널로 EC2 접속 (SSH)
      → Java 설치
        → 프로젝트 clone
          → application.yml 생성
            → 빌드 (gradlew)
              → JAR 파일 실행
```

---

### Java 설치

```bash
sudo apt update
sudo apt install openjdk-17-jdk -y
```

설치 확인
```bash
java -version
```

---

### 프로젝트 clone

```bash
git clone https://github.com/maybeags/ec2_springboot_sample.git
cd ec2_springboot_sample
```

---

### application.yml 생성

`application.properties`는 민감한 정보(client-id, DB 비밀번호 등)가 포함되어 있어  
`.gitignore`에 추가하는 것이 일반적이다. 그래서 EC2 내에서 직접 생성한다.

```bash
cd src/main/resources
vi application.yml
```

**vi 에디터 기본 명령어**

| 명령어 | 동작 |
|--------|------|
| `i` | 입력 모드 진입 |
| `ESC` | 입력 모드 종료 |
| `:wq` | 저장 후 종료 |
| `:q!` | 저장 없이 강제 종료 |

붙여넣기: `Ctrl + Shift + V`

```yml
server:
  port: 80
```

> **주의**: 들여쓰기는 스페이스 2칸. 탭 사용 불가.  
> YAML은 들여쓰기가 틀리면 실행 자체가 안된다.

내용 확인
```bash
cat application.yml
```

### 빌드 및 실행

루트 프로젝트로 이동
```bash
cd ../../..   # src/main/resources → src/main → src → 프로젝트 루트
```

현재 경로 확인
```
ubuntu@ip-172-31-13-172:~/ec2_springboot_sample$
```

빌드 실행
```bash
./gradlew clean build   # 기존 빌드 파일 삭제 후 새로운 JAR 파일 빌드
```

Permission denied 오류 발생 시
```bash
chmod +x gradlew        # 실행 권한 부여
./gradlew clean build   # 다시 빌드
```

빌드 완료 후 JAR 파일 위치로 이동
```bash
cd build/libs
ls
```

```
aws-ec2-springboot-0.0.1-SNAPSHOT-plain.jar
aws-ec2-springboot-0.0.1-SNAPSHOT.jar       ← 이 파일 실행
```

| 파일 | 설명 |
|------|------|
| `plain.jar` | 의존성 없이 순수 클래스 파일만 포함 → 실행 불가 |
| `.jar` | 의존성 포함한 실행 가능한 파일 (executable jar) |

서버 실행
```bash
sudo java -jar aws-ec2-springboot-0.0.1-SNAPSHOT.jar
```

### 성공 로그 확인 포인트

```

Started AwsEc2SpringbootApplication in 3.785 seconds  ← 정상 실행 확인
```

> 브라우저에서 `http://퍼블릭IP` 접속 시 **Whitelabel Error Page**가 뜨면 정상.  
> 서버가 실행 중이지만 매핑된 엔드포인트가 없어서 뜨는 것이다.

## 9. 탄력적 IP (Elastic IP)

### 문제점

EC2 인스턴스를 중지했다가 재시작하면 퍼블릭 IP가 변경된다.
```
중지 전: http://43.201.84.1
재시작 후: http://43.202.54.183  ← 바뀜
```

IP가 바뀌면 프론트엔드의 `.env` 파일도 매번 수정해야 한다.

### 탄력적 IP란?

**변경되지 않는 고정 IP 주소**.  
AWS 공식 정의: 동적 클라우드 컴퓨팅을 위해 고안된 정적 IPv4 주소.

### 할당 방법

1. 네트워크 및 보안 탭 → 탄력적 IP 클릭
2. 탄력적 IP 주소 할당
3. 인스턴스에 탄력적 IP 주소 연결

> **주의**: 탄력적 IP는 EC2 인스턴스에 연결되어 있을 때는 무료.  
> 인스턴스 없이 단독으로 보유만 하고 있으면 비용이 발생한다.

---

## 10. EC2 삭제 방법 (비용 방지)

반드시 아래 순서로 삭제해야 비용이 발생하지 않는다.

```
1. 인스턴스 중지
2. 탄력적 IP Release (연결 해제 후 삭제)
3. 인스턴스 종료 (삭제)
```

> 탄력적 IP를 먼저 Release하지 않으면 비용이 계속 발생할 수 있다.

## 11. 자주 발생하는 오류 & 실수 모음

### 리전 관련

**기본 리전이 버지니아 북부로 설정된 경우**
- AWS 콘솔 접속 시 항상 리전을 서울(`ap-northeast-2`)로 변경했는지 확인할 것.
- 다른 리전에 만들어진 인스턴스는 서울 리전에서 보이지 않는다.

---

### application.yml 관련

**들여쓰기 오류**
```yml
# ❌ 탭 사용 또는 공백 개수 불일치
server:
	port: 80      # 탭 사용 → 오류

# ✅ 스페이스 2칸
server:
  port: 80
```

**파일 경로 오류**
```bash
# ❌ 잘못된 경로에 생성
vi application.yml   # 루트에서 생성

# ✅ 반드시 src/main/resources 안에 생성
cd src/main/resources
vi application.yml
```

---

### 빌드 관련

**Permission denied**
```bash
# ❌ 권한 없이 실행
./gradlew clean build

# ✅ 권한 부여 후 실행
chmod +x gradlew
./gradlew clean build
```

**`cd/build/libs` 경로 오류**
```bash
# ❌ 슬래시 앞 공백 누락
cd/build/libs

# ✅ 올바른 명령어
cd build/libs
```

**plain.jar 실행 시도**
```bash
# ❌ plain.jar는 의존성 없는 파일 → 실행 불가
sudo java -jar aws-ec2-springboot-0.0.1-SNAPSHOT-plain.jar

# ✅ 의존성 포함된 파일 실행
sudo java -jar aws-ec2-springboot-0.0.1-SNAPSHOT.jar
```

---

### 탄력적 IP 관련

**인스턴스 삭제 전 탄력적 IP Release 누락**
- 탄력적 IP가 인스턴스 없이 단독으로 남아있으면 비용 발생
- 반드시 인스턴스 삭제 전에 탄력적 IP를 먼저 Release할 것
