
## 2. 도메인(Domain)

`naver.com`, `youtube.com`처럼 **문자로 만들어진 주소**.

IP 주소는 숫자로 이루어진 컴퓨터 주소인데, 사람이 외우기 어렵기 때문에  
문자열을 만들어서 IP 주소와 매핑시킨 것이 도메인이다.

```
도메인:  naver.com
IP:      223.130.195.200
→ 사람은 naver.com으로 입력하면 컴퓨터가 알아서 IP로 변환해서 접속
```

### 서브 도메인(Sub Domain)

`map.naver.com`, `search.naver.com`처럼 `[서비스명].naver.com` 형태.  
**하나의 도메인 아래에서 여러 서비스를 구분하여 관리**할 때 사용한다.

> `naver.com` 하나만 구매하면 모든 서브 도메인을 무료로 사용 가능.  
> 서브 도메인마다 별도로 구매하지 않아도 된다.

### 실무에서의 서브 도메인 활용

`example.kr` 도메인을 구매했다고 가정하면:

| 서브 도메인 | 용도 |
|------------|------|
| `example.kr` | 메인 웹사이트 (프론트엔드) |
| `admin.example.kr` | 관리자용 웹사이트 |
| `api.example.kr` | 백엔드 API 서버 |

---

## 3. 도메인을 적용하는 이유

**이유 1. 기억하기 쉬움**  
숫자로 된 IP 주소(`43.201.84.1`)보다 문자열 주소(`api.example.kr`)가 직관적이다.

**이유 2. HTTPS 적용 (더 중요한 이유)**  
일반적인 IP 주소로는 HTTPS를 적용할 수 없다.  
실무에서 서비스를 운영할 때는 도메인 적용이 사실상 필수적이다.

## 4. DNS (Domain Name System)

**도메인 주소를 IP 주소로 변환하는 시스템.**

사람은 문자열 주소를 더 잘 외우고, 컴퓨터는 IP 주소를 더 잘 처리하므로  
중간에서 변환해주는 체계가 필요한데, 이것이 DNS이다.

```
사용자가 브라우저에 naver.com 입력
  → DNS 서버에 naver.com의 IP 주소 요청
    → DNS 서버가 223.130.195.200 반환
      → 브라우저가 해당 IP로 접속
```

### DNS 레코드 종류

| 레코드 | 설명 | 사용 예시 |
|--------|------|----------|
| **A** | 도메인을 IPv4 주소로 연결 | `api.example.kr → 43.201.84.1` |
| CNAME | 도메인을 다른 도메인으로 연결 | `www.example.kr → example.kr` |
| MX | 이메일 서버 지정 | Gmail 등 이메일 서비스 연동 |
| TXT | 도메인 소유 확인 등 텍스트 정보 | Google Search Console 인증 등 |

> 이번 수업에서는 서브 도메인에 **A 레코드**를 적용하여  
> 백엔드 서버의 탄력적 IP와 연결했다.  
> 참고: https://cloud-information.tistory.com/11

## 5. HTTP vs HTTPS

대부분의 웹 사이트는 **HyperText Transfer Protocol(HTTP)** 방식으로 서버와 데이터를 주고받는다.  
그 중계 역할을 하는 데이터 형식이 JSON이다.

**HTTP의 문제점**: 주고받는 데이터를 암호화하지 않기 때문에  
중간에 데이터를 가로채는 것이 가능하다.

```
예시: 로그인 시 아이디/비밀번호를 백엔드로 보낼 때
HTTP → 평문 그대로 전송 → 해커가 가로챌 수 있음
HTTPS → 암호화해서 전송 → 가로채도 해독 불가
```

| 구분 | HTTP | HTTPS |
|------|------|-------|
| 암호화 | ❌ | ✅ |
| 포트 | 80 | 443 |
| 주소창 | 경고 표시 | 자물쇠 아이콘 |
| 보안 | 데이터 평문 전송 | SSL/TLS 암호화 전송 |

### HTTPS 적용 이유

1. **보안 강화** — FE/BE 서버 모두 HTTPS 적용 필요
2. **SEO** — 구글, 네이버 등 검색 엔진에서 HTTPS 적용 사이트를 상위 노출
3. **사용자 이탈 방지** — HTTPS 미적용 시 Chrome 주소창에 경고 표시

## 6. ELB (Elastic Load Balancing)

AWS에서 제공하는 **로드 밸런서 서비스**.

로드 밸런서란 **트래픽을 여러 서버에 분산하는 장치**로,  
특정 서버에 트래픽이 집중되는 것을 방지하고  
장애가 발생해도 정상 서버로 트래픽을 전달할 수 있도록 한다.

> 같은 역할을 하는 서버를 2대 이상 운영하는 경우 ELB를 도입한다.  
> ELB에서 HTTPS 요청을 처리하도록 설정할 수 있어  
> 보안이 필요한 웹사이트나 API 서버에서도 많이 사용한다.

### ELB 구성요소

**1. 리스너(Listener)**
- ELB로 들어오는 요청을 어떻게 처리할지 결정하는 규칙
- 특정 포트와 프로토콜로 클라이언트 요청을 기다리다가 대상 그룹으로 전달
- HTTPS 리스너는 443 포트에서 보안 연결을 통해 들어오는 트래픽을 처리
- **리스너 설정이 잘못되면 요청이 올바른 대상 그룹으로 전달되지 않으므로 주의**

**2. 대상 그룹(Target Group)**
- ELB가 수신한 트래픽을 전달할 서버들의 집합
- EC2 인스턴스를 추가하면 ELB가 해당 인스턴스로 요청을 전달
- **상태 검사(Health Check)**: 주기적으로 인스턴스에 요청을 보내 살아있는지 확인
  - `200 OK` 반환 → 정상 → 요청 전달
  - `200 OK` 미반환 → 비정상 → 해당 인스턴스에 요청 전달 안 함

```
클라이언트 요청 (HTTPS 443)
  → ELB 리스너
    → 대상 그룹 확인 (Health Check)
      → 정상 EC2 인스턴스로 트래픽 전달
        → Spring Boot (8080)
```

## 7. Nginx

웹 서버이자 **리버스 프록시 서버**.

손님(사용자)과 주방(Spring Boot) 사이에서  
주문을 효율적으로 관리하고 서빙하는 역할.

### 사용 이유

| 기능 | 설명 |
|------|------|
| **리버스 프록시** | 사용자가 실제 서버(8080)에 직접 접근 못하게 막고 대신 받아서 전달 → 보안 강화 |
| **로드 밸런싱** | 접속자 폭주 시 여러 서버로 요청 분산, 서버 다운 시 다른 서버로 자동 전환 |
| **정적 파일 처리** | HTML/CSS/이미지 등 정적 파일을 Spring Boot 거치지 않고 직접 응답 → 효율 향상 |
| **SSL(HTTPS) 설정** | 보안 인증 처리를 Nginx 내에서 처리 가능 |

### Apache vs Nginx

| 구분 | Apache (Tomcat) | Nginx |
|------|----------------|-------|
| 방식 | 동기 방식 | **비동기 이벤트 기반** |
| Spring Boot 기본 | ✅ | ❌ (별도 설치) |
| 최근 채용 추세 | 여전히 사용 | 점점 증가 |

> Spring Boot의 기본 웹 서버는 여전히 Apache(Tomcat)이지만,  
> 최근에는 앞단에 Nginx를 두는 방식을 많이 채용한다.

## 8. Nginx 설치 및 설정

> **주의**: Nginx가 80 포트를 점유하므로  
> Spring Boot의 `application.yml` 포트를 **8080으로 변경**해야 한다.

### 설치 및 확인

```bash
sudo apt install -y nginx    # Nginx 설치
nginx -v                     # 설치 확인
```

### 설정 파일 생성

```bash
# /etc/nginx/sites-available/springboot 파일을 편집기로 생성
sudo nano /etc/nginx/sites-available/springboot
```

**명령어 해석**
- `sudo` — 관리자 권한으로 실행
- `nano` — 리눅스 텍스트 편집기 (vi보다 사용하기 쉬움)
- `/etc/nginx/sites-available/springboot` — 해당 경로에 springboot 파일 생성/수정

**nano 편집기 기본 단축키**

| 단축키 | 동작 |
|--------|------|
| 텍스트 입력 | 바로 입력 가능 (vi처럼 모드 전환 불필요) |
| `Ctrl + X` | 저장 여부 확인 |
| `Y` | 저장 확인 |
| `Enter` | 파일명 확인 후 종료 |
| 붙여넣기 | `Ctrl + Shift + V` |

### Nginx 설정 파일 내용

```nginx
server {
  listen 80;
  server_name api.도메인주소;

  location / {
    proxy_pass http://localhost:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

| 설정 항목 | 설명 |
|----------|------|
| `listen 80` | 80 포트로 들어오는 요청 수신 |
| `server_name` | 이 설정을 적용할 도메인 주소 |
| `proxy_pass` | 요청을 전달할 실제 서버 (Spring Boot 8080) |
| `proxy_set_header Host` | 원본 호스트 정보를 백엔드로 전달 |
| `proxy_set_header X-Real-IP` | 실제 클라이언트 IP를 백엔드로 전달 |
| `proxy_set_header X-Forwarded-For` | 프록시를 거친 IP 목록 전달 |
| `proxy_set_header X-Forwarded-Proto` | 원본 프로토콜(http/https) 전달 |

### 설정 적용

```bash
# 1. sites-available → sites-enabled 심볼릭 링크(바로가기) 생성
sudo ln -s /etc/nginx/sites-available/springboot /etc/nginx/sites-enabled/

# 2. 기존 기본 설정 파일 삭제
sudo rm -f /etc/nginx/sites-enabled/default

# 3. 문법 오류 확인 후 Nginx 재시작
sudo nginx -t && sudo systemctl restart nginx
```

**각 명령어 설명**

1. **심볼릭 링크 생성 이유**  
   Nginx는 `sites-available`의 파일을 직접 실행하지 못하고  
   `sites-enabled`에 있는 파일만 읽어서 서버를 구동한다.  
   원본은 `sites-available`에 보관하고, 필요할 때만 바로가기(`sites-enabled`)를 통해 설정을 적용한다.

2. **default 파일 삭제 이유**  
   `sites-enabled/default`는 Nginx 기본 설정 파일이다.  
   이것을 삭제해야 우리가 만든 `springboot` 설정이 적용된다.

3. **`nginx -t` 이유**  
   Nginx 설정 파일에 오타나 문법 오류가 있으면 재시작 시 Nginx 전체가 죽을 수 있다.  
   재시작 전에 항상 문법 테스트를 먼저 실행한다.

**성공 메시지**
```
nginx: configuration file /etc/nginx/nginx.conf test is successful
```

## 9. HTTPS 적용 (Certbot)

**Certbot**: 무료 SSL 인증서를 발급해주고 Nginx 설정을 자동으로 HTTPS로 수정해주는 프로그램.

```bash
# 1. snap 패키지 매니저로 Certbot 설치
sudo snap install --classic certbot

# 2. 어느 경로에서든 certbot 명령어를 사용할 수 있도록 바로가기 생성
sudo ln -s /snap/bin/certbot /usr/bin/certbot

# 3. SSL 인증서 발급 및 Nginx 설정 자동 수정
sudo certbot --nginx -d api.도메인주소
```

**각 명령어 설명**

| 옵션 | 설명 |
|------|------|
| `--classic` | 시스템 접근 권한 부여 |
| `sudo ln -s` | `/snap/bin/certbot`을 `/usr/bin/certbot`으로 바로가기 생성. 어느 경로에서든 `certbot` 명령어 실행 가능 |
| `--nginx` | Nginx 설정 파일을 자동으로 HTTPS로 수정 |
| `-d 도메인주소` | 인증서를 발급받을 도메인 지정 |

> **인증서 발급 성공 시**: `https://` 주소로 접속 가능해진다.  
> **프론트엔드**는 정적 페이지 배포를 통해 별개의 도메인을 발급받는다.

## 11. 자주 발생하는 오류 & 실수 모음

### Nginx 설정 관련

**Nginx 설치 후 Spring Boot 포트를 8080으로 변경하지 않은 경우**
```yaml
# ❌ Nginx와 Spring Boot 둘 다 80 포트 사용 → 충돌
server:
  port: 80

# ✅ Nginx가 80을 점유하므로 Spring Boot는 8080으로 변경
server:
  port: 8080
```

**`/etc/ngnix/` 오타**
```bash
# ❌ nginx 오타 → 경로 없음 오류
sudo ln -s /etc/nginx/sites-available/springboot /etc/ngnix/sites-enabled/

# ✅ 정확한 경로
sudo ln -s /etc/nginx/sites-available/springboot /etc/nginx/sites-enabled/
```

**심볼릭 링크 없이 Nginx 재시작**
```bash
# ❌ 바로가기 없으면 설정 적용 안됨
sudo systemctl restart nginx

# ✅ 반드시 심볼릭 링크 생성 후 재시작
sudo ln -s /etc/nginx/sites-available/springboot /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl restart nginx
```

**default 파일 미삭제**
```bash
# ❌ default가 남아있으면 springboot 설정보다 default가 우선 적용될 수 있음
# ✅ 반드시 삭제
sudo rm -f /etc/nginx/sites-enabled/default
```

**Nginx 재시작 전 문법 테스트 생략**
```bash
# ❌ 문법 오류 있을 때 바로 재시작하면 Nginx 전체가 죽을 수 있음
sudo systemctl restart nginx

# ✅ 항상 테스트 먼저
sudo nginx -t && sudo systemctl restart nginx
```

### Certbot 관련

**DNS A 레코드 미연결 상태에서 인증서 발급 시도**
```
# 도메인이 탄력적 IP를 가리키지 않으면 인증서 발급 실패
# ✅ 내도메인.한국 등에서 A 레코드로 탄력적 IP 연결 확인 후 발급
```

**`-d` 옵션에 `http://` 포함**
```bash
# ❌ http:// 붙이면 오류 발생
sudo certbot --nginx -d http://api.도메인주소/

# ✅ 도메인 주소만 입력
sudo certbot --nginx -d api.도메인주소
```

---

### 보안 그룹 관련

**HTTPS 적용 후 443 포트 미추가**

Certbot으로 HTTPS를 적용했어도 보안 그룹에 443 포트가 없으면 외부 접근이 차단된다.

| 포트 | 용도 | 필요 여부 |
|------|------|----------|
| 22 | SSH 접속 | 필수 |
| 80 | HTTP | 필수 |
| 443 | HTTPS (Certbot 적용 후) | **추가 필요** |
| 8080 | Spring Boot 내부 통신 | 외부 오픈 불필요 |

## 오늘 학습 요약

1. **도메인** — IP를 문자열로 매핑. 서브 도메인으로 서비스 구분 (api / admin 등)
2. **DNS** — 도메인 → IP 변환 시스템. A 레코드로 탄력적 IP와 연결
3. **HTTP vs HTTPS** — 암호화 여부 차이. IP 주소로는 HTTPS 적용 불가
4. **ELB** — 트래픽 분산 서비스. 리스너(수신) + 대상 그룹(전달) + Health Check
5. **Nginx** — 리버스 프록시. 80 포트 점유 → Spring Boot 포트 8080으로 변경 필수
6. **심볼릭 링크** — `sites-available`(원본) → `sites-enabled`(바로가기) 구조
7. **Certbot** — 무료 SSL 인증서 자동 발급 + Nginx 설정 자동 수정
8. **보안 그룹** — HTTPS 적용 후 443 포트 추가 필수
9. **다음 수업** — 백그라운드 실행으로 터미널 종료 후에도 서버 유지
